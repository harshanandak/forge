# Forge v3 — Template Library & (Optional) Community Merge Flow

**Thesis:** Forge must deliver value at **N=1**. A single user, alone, with no
community, gets working scaffolds out of the box. Templates are the day-one
product. Publishing, marketplace, and upstream-merge are **optional appendix
infrastructure** that activates *if* a community forms — not a load-bearing
part of the value proposition.

---

## DAY-ONE VALUE (the only thing that has to ship)

A user runs `forge new adapter-pr-reviewer --name coderabbit`, answers three
prompts, gets a wired-up adapter directory with passing red-bar tests,
edits the TODO markers, runs `forge add ./coderabbit`, and now `/review`
calls CodeRabbit. **No GitHub account, no marketplace, no other users
required.** Templates ship in-tree with Forge core; they work offline.

---

## 1. TEMPLATE LIBRARY structure (in-repo, ships with core)

Templates live at `templates/<name>/` inside the Forge repo. Pinned to the
Forge version the user installed — upgrades arrive via `forge upgrade`, not
a network registry. Discovery: `forge new --list` reads
`templates/index.json` (generated by `scripts/sync-templates.js`).

**Five shipped templates (the day-one library):**

| Name | Purpose |
|---|---|
| `adapter-issue-tracker` | Skeleton for an issue-tracker adapter (Linear/Jira/Asana shape, mirrors Beads adapter) |
| `adapter-pr-reviewer` | Skeleton for a PR-review adapter (CodeRabbit/Greptile/SonarCloud shape) |
| `stage-custom` | Skeleton for a new stage block (`defineStage()` + gates + tests) |
| `gate-quality-check` | Skeleton for a quality gate that plugs into `/validate` or `/ship` |
| `skill-superpower` | Skeleton for a skill block (markdown + frontmatter + optional script) |

These exist whether or not a single other Forge user ever appears.

## 2. TEMPLATE FORMAT — `template.yaml`

```yaml
apiVersion: forge.dev/v1
kind: Template
metadata:
  name: adapter-pr-reviewer
  description: Scaffold a PR-review adapter (CodeRabbit/Greptile-style)
  category: adapter
spec:
  prerequisites:
    forge: ">=3.0.0"
    tools: [gh]
  params:
    - { name: vendor,    prompt: "Vendor slug (lowercase)", required: true }
    - { name: api_base,  prompt: "API base URL",            default: "https://api.example.com" }
    - { name: auth_kind, prompt: "Auth (token|oauth)",      choices: [token, oauth] }
  files:
    - from: scaffold/adapter.js.tmpl      to: lib/adapters/{{vendor}}/index.js
    - from: scaffold/adapter.test.js.tmpl to: lib/adapters/{{vendor}}/index.test.js
    - from: scaffold/extension.yaml.tmpl  to: extension.yaml
    - from: scaffold/README.md.tmpl       to: README.md
  variants:
    - { name: minimal,    skip: [README.md] }
    - { name: with-cache, extraFiles: [scaffold/cache.js.tmpl] }
  postInit: ["forge test", "git init && git add -A"]
```

## 3. `forge new <template>` flow (the day-one experience)

```
$ forge new adapter-pr-reviewer --name coderabbit-adapter
? Vendor slug (lowercase): coderabbit
? API base URL [https://api.example.com]: https://api.coderabbit.ai
? Auth (token|oauth): token
? Variant (minimal|with-cache) [with-cache]:

→ Resolving template: templates/adapter-pr-reviewer (v3.0.0, local)
→ Rendering 4 files into ./coderabbit-adapter/
→ Running postInit:
    forge test            ✓ 3/3 passing (red-bar placeholders)
    git init              ✓ initialized
→ Wrote .forge/template-state.yaml (parent: forge:adapter-pr-reviewer@3.0.0)

Next steps:
  cd coderabbit-adapter
  edit lib/adapters/coderabbit/index.js   # TODO markers
  forge add ./                            # install into your Forge project
```

`.forge/template-state.yaml` records `parent` + `parentSha` + files-touched.
Used locally for `forge upgrade` three-way merges when Forge ships a newer
template version. Useful even with N=1: the user can pull template
improvements without losing their edits.

## OPTIONAL APPENDIX — sections 4–10 only matter if a community forms

Everything below is **bonus infrastructure**. None of it blocks day-one
value. Treat as "if/when community happens, this is how it works."
Implementation order: ship sections 1–3 first, observe whether anyone
publishes anything, only then build sections 4–10.

### 4. PUBLISHING flow (if/when)

`forge publish` validates `extension.yaml`, runs conformance, bumps semver,
tags, and `gh release create`s. Channel = GitHub releases + a JSON file. No
bespoke server. `forge marketplace submit` opens a PR against
`anthropics/forge-marketplace.json` adding one entry
(`{ name, source: "gh:user/repo", ref, sha, tier: "community", author }`).
Trust gate: CI workflow `marketplace-conformance.yml` clones at `sha`, runs
`forge conformance run` (manifest schema, sandbox install, declared-vs-used
permissions, no postInstall network calls). Same plumbing Claude Code
marketplaces already use (`ref` + `sha` dual pin, `strict` mode).

### 5. UPSTREAM-MERGE flow (if/when)

Three mechanisms, all operating on a published GitHub block:

- **A. Promote to community tier:** `forge marketplace submit` PR; CI green;
  maintainer merges. Block resolvable as `forge add @user/name`.
- **B. Promote to official tier:** PR moves entry from
  `forge-community.json` to `forge-official.json`. Bar: 30-day soak,
  ≥10 installs, no open CVEs, two maintainer approvals.
- **C. Absorb into default kit:** `forge upstream-pull <name>` clones to
  `.upstream-staging/`, `forge upstream-diff` against the closest default,
  maintainer hand-merges into `templates/` or `lib/stages/` preserving
  `Co-Authored-By:` and `derived-from:`. Original marketplace entry gets
  `superseded_by: core@<version>`; existing installs keep working.

Trust gate: conformance CI green + maintainer approval (+ DCO signoff for C).

### 6. VERSIONING & FORKING (if/when)

`forge.lock` pins `ref` + `sha`. `forge update` bumps within `peerRange`.
`forge upstream-status` lists installed blocks whose parent has new commits
since recorded `parentSha`. `forge upstream-merge <name>` is a three-way
merge (user fork, ancestor, upstream). Reverse direction:
`forge upstream-propose` opens a PR from user's fork to the original
repo, body pre-filled from diff + template-state history.

### 7. DISCOVERY (if/when)

`forge search <query>` against locally-cached `forge-marketplace.json`
(`--tier`, `--kind` filters). GitHub topic `forge-block` as free secondary
index. Static Pages site rendered from the JSON. `forge new --list`
remains local-only and works offline at N=1.

### 8. INCENTIVE STRUCTURE (if/when)

`author` field surfaced in `forge add` output. Marketplace entry shows
install count, ★, `Featured`, `Promoted to core`. Promotion preserves
attribution forever via `derived-from:`. Quarterly "Block of the quarter"
in release notes. None of this exists or matters at N=1.

### 9. CONFLICT RESOLUTION (if/when)

Author-namespaced names per `extension-system.md` §3
(`@harsha/coderabbit-adapter` vs `@acme/coderabbit-adapter`). Bare
`coderabbit-adapter` resolves only to the official tier (zero or one entry).
Marketplace PR rejects duplicate `(@author, name)` pairs.

### 10. EXAMPLE TRACE — Maya → Bob → Forge v3.1 (illustrative, not required)

```
# Maya — same as N=1 user, just with a public push
forge new stage-custom --name coderabbit-review
edit coderabbit-review/lib/stages/review.js
forge test && gh repo create maya/coderabbit-review --public --push
forge publish && forge marketplace submit            # OPTIONAL

# Bob — only possible if marketplace exists
forge add @maya/coderabbit-review

# Forge maintainers — only if absorption is worthwhile
forge upstream-pull @maya/coderabbit-review
forge upstream-diff @maya/coderabbit-review --against templates/stage-custom
# Hand-merge into templates/stage-coderabbit/ with
#   derived-from: @maya/coderabbit-review@v1.0.0
#   Co-Authored-By: Maya <…>
gh pr create --title "v3.1: official CodeRabbit template (from @maya)"
# Marketplace entry gets superseded_by: core@3.1
```

The user clarification: this entire trace is conditional. Maya may be the
only user. The day-one library still works for her.

## 3 NEW BEADS ISSUES (priority order matches the N=1-first thesis)

1. **`forge templates: ship template library + `forge new` command (DAY-ONE)`**
   — `templates/` directory, `template.yaml` schema,
   `scripts/sync-templates.js`, `lib/commands/new.js`,
   `.forge/template-state.yaml`, and the 5 starter templates from §1.
   Required for v3.0 — solo-user value.
2. **`forge templates: local upgrade & three-way merge (DAY-ONE+)`** —
   `forge upgrade-template <dir>` does a three-way merge between
   user-edited fork, recorded `parentSha`, and the new in-tree template
   shipped with a Forge upgrade. Useful at N=1 — no marketplace needed.
3. **`forge marketplace: deferred — submission CI + conformance + upstream
   tooling (BONUS)`** — `marketplace-conformance.yml`, `forge marketplace
   submit`, `forge-marketplace.json` schema with tier/superseded_by, plus
   `forge upstream-status / upstream-pull / upstream-merge /
   upstream-propose / upstream-diff`. Build only after at least one
   external user publishes a block. Track here so it isn't forgotten,
   but explicitly **not** v3.0 scope.
