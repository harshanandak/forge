## Wave 1 (independent — can run in parallel)

### Task 1: Add rebase step to /validate entry HARD-GATE

**File(s)**: `.claude/commands/validate.md`
**What to implement**: Insert a rebase section as the FIRST step in the existing HARD-GATE block, before any validation checks run. The rebase section must:
1. `git fetch origin master` — get latest remote state
2. Check if branch is behind origin/master using `git rev-list --count HEAD..origin/master`
3. If behind (count > 0): run `git rebase origin/master`
4. If rebase succeeds: print "Rebased onto latest master (N commits)" and continue to checks
5. If rebase conflicts: run `git rebase --abort`, print conflicting files, tell user to resolve manually and re-run /validate. STOP — do NOT proceed to checks.
6. If already current (count = 0): print "Branch is up-to-date with master" and continue
7. If `git fetch` fails (no network/no remote): abort with clear error, do NOT silently skip

**Why this matters**: Without this, /validate checks code that doesn't include upstream changes. Integration issues surface only after PR creation, wasting CI cycles and review time.

**TDD steps**:
1. This is a markdown instruction file — no unit tests. Verification is manual + command eval.
2. Review the modified file for completeness: all 5 edge cases from design doc are covered
3. Commit: `feat: add rebase step to /validate entry gate`

### Task 2: Add freshness safety check to /ship pre-push

**File(s)**: `.claude/commands/ship.md`
**What to implement**: Insert a freshness check as a new step between the current "Step 1: Verify /validate Passed" and "Step 3: Push Branch". The check must:
1. `git fetch origin master` — get latest remote state
2. `git rev-list --count HEAD..origin/master` — check if behind
3. If behind: STOP and alert user: "Master has advanced since /validate. Run /validate again to rebase and re-check."
4. If current: continue to push
5. If fetch fails: abort with clear error

**Why this matters**: Time can pass between /validate and /ship (user reviews design doc, takes a break). This lightweight check catches staleness without doing a full rebase — it just alerts.

**TDD steps**:
1. Markdown instruction file — no unit tests
2. Review: freshness check is placed before push, not after
3. Commit: `feat: add freshness check to /ship pre-push`

### Task 3: Replace PR template in /ship

**File(s)**: `.claude/commands/ship.md`
**What to implement**: Replace the current PR template in "Step 4: Create PR" with the new narrative structure:

**Visible section** (always shown in PR):
- `## Problem` — What was broken / what need existed
- `## Root Cause` — Why it happened / why it was missing
- `## Fix` — What this PR does to solve it
- `## Value` — Impact: who benefits, what improves, what risk is removed
- `## Beads` — Closes: <issue-id>

**Collapsible section** (`<details><summary>Implementation Details</summary>`):
- Test Coverage (unit count, scenarios)
- Security Review (OWASP summary)
- Design Doc link
- Key Decisions from design doc
- Documentation Updated (list of docs changed)

**Why this matters**: Current template buries the "why" under implementation noise. Reviewers want to quickly understand the purpose before diving into code. The collapsible section keeps details available without cluttering the summary.

**Edge case**: If no Beads issue exists (hotfix, external contribution), skip the "Closes" line.

**TDD steps**:
1. Markdown instruction file — no unit tests
2. Review: template has all 5 visible sections + collapsible details
3. Verify the `gh pr create` heredoc syntax is correct (nested details tags can be tricky)
4. Commit: `feat: improve PR template with narrative structure`
